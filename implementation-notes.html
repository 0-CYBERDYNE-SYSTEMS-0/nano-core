1. Evaluator delivery boundary Removed user-visible evaluator verdict formatting from chat/heartbeat/scheduled/cron paths. 2. Blocking evaluation escalation Added a user-safe escalation message when blocking chat evaluation fails and auto-repair cannot proceed, exhausts retries, or repair fails. The message says the task could not be verified and asks for approval/follow-up without exposing evaluator score, issues, or feedback. This is a conservative first step toward host-owned approval. It does not yet resume automatically from an approval button because the existing callback surface is specialized for coder and Telegram group approvals. I avoided inventing an unsafe generic callback until the run-state model is tightened. node --import tsx --test tests/evaluator.test.ts passed, including assertions that escalation text excludes evaluator internals.
3. Quiet skill maintenance isolation Quiet skill maintenance now runs with a synthetic maintenance:<group> chat identity, no agent delegation tool, explicit "do not edit skill files directly" guidance, and a sandbox allowed-path override limited to the group IPC directory. Because FFT_NANO_SANDBOX can be none, path enforcement is only hard when bwrap/docker sandboxing is enabled. I kept toolMode: full so the agent can still write IPC action JSON; read_only would also disable the intended skill-action workflow. npm run typecheck passed. Static check confirmed sandboxAllowedPathsOverride, synthetic maintenance chat identity, and removal of the default agent tool via toolMode: full.
4. Durable IPC action audit Added attachActionRequestAudit() and wrapped action result writes so action_results/<requestId>.json now contains an audit.ipcRequest copy of the original request plus source group metadata. This keeps the existing top-level result shape intact for agents that check status/result, while adding proof data for validators. I did not retain consumed files in actions/ because that would risk reprocessing; the result file is the safer durable ledger. node --import tsx --test tests/action-result-audit.test.ts passed. npm run typecheck initially caught too-narrow helper types; after fixing those, typecheck passed.
5. Heartbeat private checklist Added heartbeat/checks/<requestId>.json artifacts under the main workspace for each heartbeat run. The checklist records heartbeat file presence, task snapshot presence, runtime log presence, today's memory file presence, and the host-classified outcome. This creates durable proof without making Telegram heartbeat output verbose. The evaluator is not yet fully rewired to grade only this artifact, so this is a proof foundation rather than the whole heartbeat scoring redesign. node --import tsx --test tests/heartbeat-checklist.test.ts passed. npm run typecheck passed.
6. Personal skill validation/sync Added npm run validate:personal-skills, relaxed compatible frontmatter parsing for personal skills, accepted common skill metadata fields, allowed nested metadata, and ignored embedded dependency directories such as .venv during safety scanning. I fixed this at the validator/sync layer rather than editing personal skill files. That avoids mutating user-owned skills and makes the runtime tolerant of common skill formats while still rejecting unsafe symlinks outside ignored implementation directories. node --import tsx --test tests/pi-skills.test.ts passed. npm run typecheck passed. npm run validate:personal-skills initially reproduced the six invalid skills, then passed after the parser/safety changes.
7. Recent conversation filtering Added filtering for legacy internal assistant-history rows during prompt assembly. Old evaluator/quality-check text, heartbeat acknowledgements, and empty-output diagnostics are excluded from [RECENT CONVERSATION]. This is pattern-based because the database rows do not yet carry source metadata. It fixes existing contaminated history without a migration, but a future schema-level source field would be cleaner. node --import tsx --test tests/message-dispatch.test.ts passed with a new regression for legacy evaluator rows. npm run typecheck passed.
8. Memory cleanup-claim verification Added a postcondition verifier for memory entries that claim stale skill directories were removed. Heartbeat checklist artifacts now include memoryClaims with claimed directories and any paths that still exist. This does not rewrite user memory automatically. It creates host-observed contradiction evidence first, which is safer than silently editing memory or deleting directories after a model-authored claim. node --import tsx --test tests/memory-claim-verifier.test.ts tests/heartbeat-checklist.test.ts passed. npm run typecheck passed.
